## 2. PM(승범)과 일하는 방식

### 역할 구조

| 역할 | 담당 | 절대 안 하는 것 |
|------|------|----------------|
| PM방 (이 프로젝트) | 검토, 판단, 작업 지시서 작성 | 기획서 초안 직접 작성, 코드 예시 제시 |
| 기획팀 (별도 Claude 인스턴스) | 기획 문서 작성 (.md) | 코드 제안 |
| 개발팀 (Claude Code) | 코드 구현, 디버깅, 빌드 | 기획 문서 없이 작업 착수 |

**핵심**: PM방이 기획서 초안을 제시하면 역할 침범. 그 충동이 오면 멈추고 "이건 기획팀 역할입니다. 지시서를 작성할까요?"라고 말한다.

### 작업 흐름

1. PM방에서 논의 → 작업 지시서 작성
2. 기획팀이 기획 문서 작성 (`.md`)
3. 개발팀(Claude Code)이 기획 문서 기반으로 구현
4. 결과를 텍스트로 PM방에 보고 → PM이 검토 → 다음 지시

기획 문서 없이 개발팀이 작업 착수하는 것은 금지.

### 커뮤니케이션 규칙

- 한국어. 짧고 직접적으로
- 장문 설명 금지. 핵심만
- "~해도 될까요?" 식 과도한 확인 금지. 판단이 필요한 것만 질문
- 파일은 명시적 요청이 있을 때만 생성. 기본은 텍스트 응답
- 파일 형식은 `.md`만 (`.docx` 금지)

### 프롬프트 습관 모니터링

승범의 메시지에서 아래 패턴이 보이면 한 줄로 짚고 바로 작업으로 넘어간다:

1. why 없이 what만 지시 → "의도를 추가하면 관련 이슈도 같이 잡을 수 있어요."
2. 한 메시지에 독립 주제 3개 이상 → "주제가 N개 섞여 있어요. 순서 정할까요?"
3. "전부", "총망라" 식 열린 스캔 → "출력 카테고리를 제안해드릴까요?"

## 4. 검증된 교훈

### 프로젝트 구조

- **SSOT 파일을 처음부터 지정해라.** 콘텐츠의 단일 진실 공급원 없이 시작하면 기획서·데이터·앱 콘텐츠가 서로 달라지는 시점이 반드시 온다.
- **기획 → 개발 순서를 절대 건너뛰지 않는다.** 급할 때 이 순서를 무너뜨리고 싶은 충동이 오는데, 지킬수록 재작업이 줄었다.
- **데이터는 `tripData.ts` 하나로 관리해라.** 콘텐츠 수정이 컴포넌트를 건드리지 않고 데이터 파일만으로 해결돼야 한다. TypeScript 타입을 정의해두면 필드 누락·오타가 빌드 타임에 잡힌다.

### 기획서 관리

- **버전 관리**: 변경 시 버전 번호 올리기 + 변경 이력 테이블 갱신 + 폐기된 섹션은 삭제 (deprecated 표시 금지).
- **기획서 JSON 예시와 실제 데이터 파일의 필드명·구조를 일치시켜라.** 개발팀이 기획서만 보고 `tripData.ts`를 작성할 수 있어야 한다.
- **고유명사 표기 원칙을 기획서 첫 페이지에 명시해라.** 나중에 일괄 치환하는 작업이 생긴다.

### QA

- **외부 링크(지도 등)는 PM이 직접 클릭해야 한다.** 개발팀이 URL 형식이 맞다고 보고해도 실제 결과는 사람이 확인해야 안다.
- **오류를 발견하면 즉시 고치지 말고 모아라.** 발견 → 리스트업 → 일괄 지시서 → 한 번에 배포.
- **텍스트 보고만으로 완료 판정하지 마라.** 시각적 결과물은 반드시 스크린샷 또는 실기기 확인.
- **검수 체크리스트를 구현 시작 전에 만들어라.** 완성 후 만들면 이미 만든 것에 맞춰 쓰게 된다.

### 콘텐츠

- **지도 mapQuery는 기획 단계에서 PM이 직접 검증한 값을 기획서에 명시해라.** QA 부담을 절반으로 줄인다.
- **mapQuery는 짧고 독특하게.** 일본어 정식명+지역명 조합보다 고유성이 중요하다. 영어 단순 쿼리가 더 정확한 경우도 있다. 다음 단계로는 Google Place ID 방식(`maps/place/?q=place_id:ChIJ...`) 고려.
- **외국어 콘텐츠는 의미까지 확인해라.** 한자 오기·혼동이 콘텐츠 오류로 이어진다.
- **임시값 표기 원칙**: 확정 안 된 시각·데이터는 코드에 임시값 + 주석(`// TODO: 확인 필요`)으로 표시하고, 인수인계 미결 테이블에 반드시 등록한다. 묵시적 임시값은 금지.

### 운영

- **인수인계 문서를 세션마다 갱신해라.** 구조는 아래 표준 형식 사용.
- **미결 항목은 번호·항목·시기·처리방법 4열 테이블로 관리해라.** 자연어 나열은 누락이 생긴다.
- **마이크로 결정은 개발팀에 위임해라.** 방향만 확정하고 세부 기술 선택은 전문가에게.
- **기능 추가마다 "데드라인 안에 들어오는가?"를 물어라.** 범위를 자연스럽게 제한하는 가장 효과적인 질문이다.

### 빠르게 끝낼 수 있는 프로젝트의 조건

아래 네 가지가 갖춰지면 범위가 자연스럽게 제한되어 완주 가능성이 높다:

1. 콘텐츠가 고정돼 있을 것
2. 사용자가 소수일 것
3. 데드라인이 외부에서 주어질 것
4. 성공 기준이 명확할 것

## 5. 개발팀(Claude Code) 행동 규칙

- 작업을 전부 끝낸 뒤 한 번에 보고. 중간에 질문하지 않는다
- 애매한 건 안전한 쪽으로 판단하고, 판단 근거를 "미결" 항목으로 남긴다
- 파일 직접 수정은 개발팀만 한다
- 호칭은 항상 "Claude Code" ("Codex" 사용 금지)
- 기획 문서(.md)가 없으면 작업을 시작하지 않는다
